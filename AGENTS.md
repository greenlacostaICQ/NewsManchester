# News project — agent context

Greater Manchester daily digest. Runs as a GitHub Actions workflow at
08:00 Europe/London. Python handles collect, dedupe, validate, curator,
LLM rewrite, write, edit, and release gate. LLM calls (OpenAI/Gemini/Groq)
are made by the pipeline itself — not by an external agent.

## Pipeline (in order)

```
collect-digest
  → dedupe-digest
  → validate-candidates
  → curator-pass          # editorial LLM pass: drop PR/evergreen, pick lead story
  → transport-fill        # ensure checked transport coverage / synthetic transport status
  → llm-rewrite           # 5 category-specific prompts (transport/city/events/business/football)
  → write-digest
  → edit-digest
  → build-digest          # release gate → promotes draft to outgoing + writes published_facts
```

Each stage reads/writes its own `data/state/*.json` file.
`build-digest` is the only stage that promotes
`data/state/draft_digest.html` to `data/outgoing/current_digest.html`
and updates `published_facts.json` for next-day dedupe.

Sources are declared in `data/sources.toml` — set `enabled = false` to
disable a source without touching Python. Each source can carry an
optional `notes = "..."` field with editorial rationale (why it's in
the registry, what role it plays). Surfaced on `SourceDef.notes`;
used by I4 Best Source Selection and Q1/Q8 decisions.

## Operational notes

Do not add model API calls, new pipeline stages, or shell-script changes
without updating `.github/workflows/daily-digest.yml` and
`scripts/run_local_digest.py` in the same commit.

One pass per stage; if uncertain, leave Python output as-is.

1. Run `collect-digest`. Review `data/state/candidates.json` and change
   only obvious bad includes: homepage/index/category pages, non-GM items,
   wrong-source items.
2. Run `dedupe-digest`. Review only ambiguous carry-over/new edge cases.
   Do not rewrite normal deterministic decisions.
3. Run `validate-candidates`. Review for vague practical angles and weak
   `include=true` candidates.
4. Run `curator-pass`. The curator LLM sees all included candidates at
   once, drops PR/evergreen/non-GM, and marks `is_lead=true` on the top
   story. Review `data/state/curator_report.json` if lead selection looks wrong.
5. Run `transport-fill`. Review `data/state/transport_fill_report.json`
   if the transport block is empty or if TfGM/Metrolink coverage looks
   different from the live status.
6. Run `llm-rewrite`. Each candidate category gets its own prompt.
   Drop decisions from writer (no draft_line) are logged at INFO level.
7. Run `write-digest`. Review `data/state/draft_digest.html`.
   Candidate draft_line for transport / culture / football / ticket
   items to fallback prose. Each line must start with `•`, be
   self-contained, and use Telegram HTML emphasis (`<b>...</b>`) instead
   of Markdown `**...**`. Do not write source anchor HTML;
   `write-digest` attaches source links. Save the updated `draft_line`
   values back into `data/state/candidates.json`, then run
   `write-digest`.
   Full-issue volume rule:
   - After a normal broad scan, target **14-22 visible digest items**.
   - Never shrink a full scan to a "safe" mini-issue. If the final
     draft would have fewer than **12 visible items**, go back to
     `data/state/candidates.json` and repair / rewrite additional
     candidates from the already collected pool instead of sending.
   - Remove candidates only for a concrete reason: duplicate meaning,
     stale/carry-over without a new phase, non-GM item, wrong source,
     expired event/disruption, or source material too fact-thin to make
     self-contained without inventing details.
   - Do not remove a candidate merely because it is hard to phrase; if
     the facts are present, rewrite it into normal Russian.
   `draft_line` source constraint:
   - Write only from facts present in the candidate's `title`, `summary`,
     `lead`, and `practical_angle` fields. Do not add details — dates,
     prices, codes, route names, album names, promo codes, extra context —
     that are not in those fields, even if you can verify them externally.
     The candidate is the single source of truth for this step.
   `draft_line` practical-angle rule:
   - The practical angle in every `draft_line` must end with a concrete
     imperative verb directed at the reader:
     `проверьте` / `закладывайте` / `сверьте` / `уточните` /
     `не откладывайте` / `убедитесь` / `следите` / `держите в планах`.
     Vague closings such as «это важный сигнал», «стоит знать»,
     «это новый маршрут» are banned.
   `draft_line` short_actions block format:
   - For candidates with `primary_block == "short_actions"`:
     - Start the bullet with a time-of-day prefix:
       `После работы:` / `Сегодня:` / `С 09:00:` (pick the one that fits).
     - One action per bullet. No facts, no numbers, no explanation.
     - The full fact lives in another block lower in the digest.
       The short_actions line is only the action pointer.
   `draft_line` editorial bans:
   - Do not leave English words untranslated when a normal Russian
     equivalent exists:
     `councillors` -> `депутаты совета`,
     `greengrocer` -> `овощная лавка`,
     `opening` -> `открытие`,
     `baby-and-carer` -> `для родителей с младенцами`,
     `cask ale` -> `живое пиво`,
     `dining room` -> `обеденный зал`,
     `pub menu` / `pub-меню` -> `меню паба`,
     `slot` -> `временной слот`.
   - Do not explain an item with the pattern `а не X`; rewrite
     positively and directly.
   - Do not use bureaucratic filler such as `висит карточка` or
     `слот подтверждён`.
   - For concerts and ticket items, always state the date window in
     words (for example, `1–6 мая`) so the reader does not have to infer
     when the event happens.
   - For police / incident items, the first sentence must name what
     happened, not just describe the scene.
   - Do not use empty `why it matters` lines such as `это важный
     сигнал` or `это заметный кейс` without concrete practical meaning.
   Repair the candidate before rewrite if any of these are true:
   - the item has no clear subject (who acted / who was affected / what
     institution or venue is actually involved);
   - an event or ticket item has no usable date, time, or date range;
   - a police / incident item describes only the scene and still does
     not name what happened;
   - the item relies on an English term that needs a normal Russian
     equivalent or a short explanation in the text;
   - the `practical_angle` is only a vague claim such as `важный
     сигнал` or `заметный кейс` without concrete meaning for the reader.
   Remove the candidate only if:
   - it duplicates another included candidate in the same block by
     meaning, even if the wording differs; or
   - the source material is too fact-thin to repair into a clear
     self-contained item without inventing details.
5. Run `write-digest`, `edit-digest`, and `build-digest` normally.

## Current blockers (hand-maintained — last reviewed 2026-04-28 AM)

Phase 1 is considered complete. This section tracks accepted waivers and
the current operational shape of the project, not open implementation
blockers.

- **Phase 1 closed with 2 accepted waivers.** `ITV Granada` is out of
  scope until Phase 2 because Cloudflare bot-protection defeats `urllib`
  fetches. `ManchesterWorld` is out of scope until Phase 2 because the
  public RSS surface returns affiliate/recommended commerce items rather
  than usable local-news coverage. Both are intentionally commented out in
  `collector/sources.py`.
- **LLM rewrite is external operational policy, not Python runtime.**
  Python stages never call model APIs. After `validate-candidates`, the
  agent must write Russian `draft_line` values for every
  `include=true` candidate before `write-digest`; the digest should not
  rely on generic writer fallback prose for included items.
- **`food_openings` can legitimately be empty.** `Manchester's Finest`
  often serves listicles and monthly roundups; `_is_listicle_opening`
  correctly drops them. `0 publishable` on a given day is not itself a
  failure.
- **`Ticket Radar` is low-signal and optional when empty.** If the scan
  produces no new official ticket trigger, the block should be hidden
  rather than causing a release failure.
- **Source expansion scope is frozen.** Do not re-add Royal Exchange,
  Science and Industry Museum, or Manchester Art Gallery unless the user
  explicitly reopens source expansion. Current work is final digest
  quality and operational stability, not adding more sources.

## Conventions

- All wall-clock time uses `now_london()` / `today_london()` from
  `common.py`. Never `datetime.utcnow()` or naive `datetime.now()`.
- Candidate fingerprint = `canonical_url_identity` of `source_url`
  (scheme/host/path, no query, no www). Two URLs that point at the
  same article get the same fingerprint.
- Label `freshness_status == "fresh_24h"` semantically means
  "published since yesterday midnight London", NOT literal "<= 24h
  ago". Label kept for backward compatibility with downstream report
  fields (`fresh_last_24h_count` etc.).
- A candidate with `practical_angle` starting with `Включать только…`
  is treated as "weak / placeholder" by writer/editor. These strings
  live in `common.VAGUE_PRACTICAL_ANGLES`; check via
  `is_placeholder_practical_angle()`.
- Required output blocks live in `common.REQUIRED_BLOCKS`. Gate
  fails if any is missing or empty. `common.LOW_SIGNAL_BLOCKS`
  hide when empty.
- `published_facts.json` is updated on gate-pass (in `release.py`
  via `update_published_facts`), not on Telegram send. Idempotent
  by fingerprint.

## State files (in `data/state/`)

- `collector_report.json` — per-category broad scan health
- `candidates.json` — current scan output, mutated by dedupe/validator
- `dedupe_memory.json` — today's dedupe decisions, including
  `semantic_dedup_summary` (I1: model name, embedded/cache hit counts,
  intra_drops[], cross_day_drops[], borderline_pairs[]).
- `embedding_cache.json` — per-fingerprint cached embedding vectors
  (text-embedding-3-small, 1536d). Content-hashed, pruned after 30d.
  Persisted across runs by the daily workflow so steady-state semantic
  dedup only embeds today's new candidates.
- `candidate_validation_report.json` — validator output
- `writer_report.json` — includes `rendered_candidate_fingerprints[]`
- `editor_report.json` — line dedup + balance counts
- `release_report.json` — gate verdict + `published_facts_updated`.
  `source_status.sources[]` carries per-source `candidate_count` (raw),
  `curated_count` (include=True after curator), `rendered_count`
  (curated AND in writer's `rendered_candidate_fingerprints`).
  `source_status.counts.zero_yield` flags sources that fetched OK but
  contributed nothing to the digest. Synthetic sources (Met Office
  weather, transport reminders) appear with `category="synthetic"`.
  `synthetic_freshness` lists every synthetic candidate with its
  `data_fetched_at` timestamp and `synthetic_stale` flag — populated
  by the O2 freshness gate (refetch×2 on Met Office + Open-Meteo for
  weather; 14-day freshness window on Metrolink reminders). When all
  weather sources fail, the placeholder ships and the weather-block
  digit check downgrades from error to warning so the digest never
  blocks on a network outage.
- `published_facts.json` — cross-day history (idempotent by fingerprint)
- `last_sent_digest.html` — copy of last actually-sent digest
- `delivery_state.json`, `bot_state.json` — Telegram bot state
- `fetch_cache.json` — per-URL ETag / Last-Modified validators used by
  the collector to send conditional GETs. 304 responses are treated as
  "no new content" (source marked `not_modified=true` in
  `collector_report.json`, parsing skipped). TTL 7 days; stale entries
  are ignored. Loaded once per run via `load_fetch_cache` and flushed
  via `save_fetch_cache` in `collector/fetch.py`.

`data/outgoing/current_digest.html` — what gets sent.

## Editorial regression pack (tests/test_editorial_regression.py)

Pinned-defect golden cases. 25–30 fast unittest cases covering the
seven historical failure modes (no-date event, PR, not-GM, duplicate,
stale event/synthetic, bad HTML in draft_line, untranslated English).
Each case exercises a real deterministic predicate
(`_exclude_*` / `_apply_intra_batch_dedup` / `_draft_line_quality_errors`
/ `_is_obviously_non_gm_*` / `_is_stale_transport` / etc.) — no
fixtures on disk, no LLM calls. Adding a case for a freshly-fixed
defect: copy the live candidate fields into a new test method here,
assert the predicate that should now catch it.

Run with `PYTHONPATH=src python3 -m unittest discover -s tests`.
CI: `.github/workflows/tests.yml` runs on every push to a non-`main`
branch and every PR. `main` only runs the daily digest; tests are
intentionally out of the release path so a failing test never blocks
a send.

## Common task patterns

- "Add a source" — new `SourceDef(...)` in
  `collector/sources.py:SOURCES`; add a filter branch in
  `collector/filters.py:_is_allowed_source_link`. If the source has a
  different host than its feed (e.g. RSS on `feeds.X` vs articles on
  `Y.com`), set `allowed_hosts=("...",)`.
- "Tighten a block policy" — edit the keyword tuple in
  `collector/filters.py` (`_LISTICLE_OPENINGS_PATTERNS`,
  `_FOOTBALL_FLUFF_TOKENS`, `_CITY_WATCH_TOPICAL_KEYWORDS`) or
  `collector/routing.py` (`_TODAY_FOCUS_KEYWORDS`).
- "Improve practical_angle for topic X" — add a branch in
  `collector/summary.py:_default_practical_angle` matching keywords on
  `title.lower()` and `summary.lower()`.
- "Change what counts as fresh" —
  `collector/routing.py:_freshness_status`.
- "Change what blocks the gate" — `_validate_*` functions in
  `release.py`.
- "Promote a candidate to today_focus" —
  `collector/routing.py:_promote_to_today_focus` and
  `_today_focus_score`.
- "Fix a fetch issue (403, redirects, headers)" —
  `collector/fetch.py:_fetch_text` or `_source_fetch_headers`.
- "Add a new RSS/JSON parser shape" —
  `collector/extract.py:_extract_*_items`.

## Working efficiently

- The collector is a package, not a single file. Open the relevant
  submodule (sources / filters / routing / etc) — the largest is
  ~360 LOC. Use `rg -n` with a symbol name as anchor; don't read the
  whole package.
- Pipeline stages are independent given their state files. To verify
  one stage, run that stage alone and inspect its `*_report.json`.
- The gate is fail-closed. If `release_report.json.errors` is empty,
  the digest passes; otherwise the listed errors are the punch list.
- Before adding a new helper to a collector submodule, search for an
  existing one (`rg -n 'def _' src/news_digest/pipeline/collector/`)
  — there are already multiple date parsers, multiple title cleaners,
  multiple keyword sets. Reuse before adding.
- The old `collector.py` survives as a 13-line tombstone because the
  refactor sandbox couldn't `rm` it. Python resolves to the package;
  the file can be deleted with no functional effect.
